First: What even is SAP BTP? (The big picture – super simple)
Imagine SAP BTP as a huge cloud toolbox where companies can build, run, and connect modern apps without buying 50 servers. Inside this toolbox you get:

Databases

AI services

Workflow tools

Integration pieces

And now: generative AI copilots like Joule

Joule is SAP’s AI assistant (like a company-specific ChatGPT + Copilot) that lives inside SAP apps (S/4HANA, SuccessFactors, Ariba, etc.) but can also be extended.

Joule “skills” and “agents” – what’s the difference?
A skill = one specific superpower the AI can use
Example: “Look up a sales order status”, “Calculate delivery date”, “Call a weather API”, “Post something to Teams”

An agent = a smart boss that decides which skills to use (and in which order) to solve a bigger, multi-step problem
Example: “Help me handle a customer complaint” → agent might:

Look up order (skill A)

Check stock (skill B)

Create return request (skill C)

Send apology email (skill D)

You build skills first, then you give them to an agent so it can reason and chain them.

Now the key part: How does a skill actually “do” something useful?
Most useful skills need to talk to real systems:

SAP systems (S/4HANA, SuccessFactors…)

External APIs (your own microservice, a partner REST API, weather service, etc.)

Databases, file shares, etc.

When Joule (or rather: the agent using the skill) wants to call your REST endpoint, it can’t just magically know:

What is the URL? (https://api.yourcompany.com/v1/orders)

What authentication do I need? (Basic Auth? OAuth2 client credentials? Certificate? SAML? API key?)

Any proxy / on-premise tunnel needed? (Cloud Connector if the API is behind your firewall)

Any extra headers, certificates, trust settings?

If every skill developer had to hard-code all of this → nightmare (security holes, can’t change passwords centrally, no reuse).

So SAP invented Destinations – the “phone book” for connections
A Destination in BTP is basically a named, centrally managed connection config. Think of it as:

A contact card in your phone: “Call Mom” → behind the scenes: +90 212 555 1234, mobile, favorite

You don’t type the number every time; you just say “Call Mom”

In BTP:

You create a destination once in the BTP cockpit (under Connectivity → Destinations)

You give it a name, e.g., MY_COMPANY_ORDER_API

You fill in:

URL

Authentication type + credentials / OAuth client ID & secret / etc.

Proxy type (Internet, OnPremise via Cloud Connector…)

Additional properties if needed (e.g., sap-client, HTML5.DynamicDestination=true…)

Then anywhere in BTP (Build Process Automation, AI Core, Joule Studio, etc.) you can just reference that destination name instead of typing credentials 50 times.

Why is a destination required when you add a REST endpoint as a skill in Joule?
Because Joule Studio (where you build skills & agents) does not let you hard-code URLs + passwords directly into the skill for security and governance reasons.

Instead the flow is (typical real-world path):

You have a REST API somewhere (your own, partner’s, whatever)

You create a Destination in your BTP subaccount that points to that API (with auth)

You go to SAP Build → Action Projects (this is the bridge)

Create an “Action Project” that imports OpenAPI spec or manually defines actions

In that action project you select / reference the Destination you created

This creates a safe, governed “API wrapper” that Joule can call

In Joule Studio you create a Skill

Add an Action to the skill

The action points to the Action Project you just made

Deploy the skill

Create / edit your Agent and assign that skill to it

So when the agent needs to use your REST skill:

Joule → calls the Action Project → Action Project uses the Destination config → securely calls your real REST endpoint

No password leakage in the skill definition, central credential rotation, audit trail, etc.

Quick summary – why destination is required / not optional
Joule agents/skills are not allowed to contain raw credentials or URLs (enterprise security)

Destinations are the secure, reusable, governable way to reach any HTTP/REST endpoint

Without a destination → no place to safely store “where + how to authenticate” → can’t call the endpoint

If you’re just playing around and the REST endpoint is public/no-auth (e.g., a free test API like jsonplaceholder.typicode.com), some tutorials let you put the URL directly — but in real scenarios (especially company-internal APIs) destination is mandatory.

In SAP Build lobby → create Action project → use that destination

In Joule Studio → new skill → add action → pick the action project

Test the skill → then create agent → add the skill

Build/test the agent + its skills in DEV subaccount’s Joule Studio.

Build in DEV Joule Studio.

Export skills/agents as JSON / project bundles (available in Studio UI).

Store in Git (with destinations as placeholders / env vars).

In TEST/PROD subaccounts:

Import the bundle into Joule Studio.

Adjust destinations / configs via environment variables or manual mapping.

Deploy.

Let's explain it Feynman-style—like we're comparing two neighboring houses in the same neighborhood, one old and one brand new.

### The Old House: Classic On-Prem SAP (What You Know Best)
- You (or the customer) own the land, build the house (hardware), install the plumbing/electricity (infrastructure), paint the walls (apply support packs), and maintain everything (basis team runs upgrades, backups, monitoring).
- Your ABAP code lives inside the house walls (modifications, user exits).
- Fiori apps are like fancy windows added later.
- Pain points: Slow upgrades, hardware costs, downtime for maintenance.

### The New House: RISE with SAP (The "Rise and on Cloud" Thing)
RISE with SAP is **not** a separate product or platform—it's a **complete bundled offering** (a "transformation package") from SAP that helps move your business to the cloud **with S/4HANA as the core**.

Think of RISE as:  
**SAP saying, "Stop worrying about building and maintaining the house. We'll give you a ready-made, modern cloud house (S/4HANA Cloud), fully managed by us, plus tools to renovate it intelligently."**

Key pieces inside a RISE bundle (as of 2026):
- **SAP S/4HANA Cloud, private edition** (or public edition in some cases) → This is the main living space. Private edition feels more like your old on-prem S/4 (customizable ABAP, familiar transactions), but it's fully hosted and managed by SAP on hyperscalers (AWS, Azure, Google Cloud).
- **Infrastructure & operations** → SAP handles servers, patching, backups, high availability, disaster recovery. You don't need a big basis team anymore.
- **Business transformation services** → Tools like SAP Signavio (process intelligence), preconfigured best practices, migration tools.
- **Clean Core promise** → Extensions go outside the core (no more heavy mods).
- **Built-in credits for BTP** → Here's the link: Every RISE contract includes **some BTP credits** (like free fuel vouchers) so you can build extensions in the BTP "wing" next door.

### How RISE and BTP Fit Together (The Neighborhood View)
- **RISE** = The fully managed S/4HANA cloud house (the ERP core lives here).
- **BTP** = The flexible extension wing/garage/workshop next to it (where you build custom apps, integrations, AI stuff, Fiori extensions, ABAP cloud code).
- They are **designed to work together tightly**:
  - From inside RISE (S/4HANA Cloud), you call BTP services easily (e.g., a custom Fiori app in BTP pulls data from S/4 via secure OData).
  - Destinations in BTP subaccounts point back to the RISE S/4 system.
  - Cloud Connector (if needed) tunnels on-prem legacy stuff to both.
  - Many RISE customers get a **dedicated BTP subaccount** (or entitlements assigned) as part of the deal—often called "BTP for RISE" or "RISE extension credits."

Analogy for ABAP/Fiori devs:
- In old world: You modify the S/4 house directly (Z-tables, enhancements inside core).
- In RISE world: The S/4 house is "rented and maintained by landlord (SAP)" → you can't hammer nails into core walls anymore (Clean Core enforced more strictly).
- So you build add-ons in the **BTP garage next door**:
  - Write RAP-based ABAP in BTP ABAP Environment → expose as OData.
  - Build Fiori/UI5 apps in Business Application Studio → deploy to BTP subaccount.
  - Use Integration Suite (in BTP) for any orchestration.
  - The bridge (destinations, APIs) makes it feel like one big connected property.
